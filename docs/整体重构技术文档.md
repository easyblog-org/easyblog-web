# EasyBlog 重构技术方案 v2.0

## 一、项目背景与目标

### 1.1 现状问题

当前 EasyBlog 是一个基于 Nuxt 2 + Vuetify 的博客系统，存在以下核心问题：

| 问题 | 描述 |
|------|------|
| 后端 API 过重 | 文章数据依赖后端 API（`api.easyblog.top`）进行 CRUD，需要维护数据库和后端服务，运维成本高 |
| Vuetify 深度耦合 | 所有组件依赖 `<v-app>`、`<v-card>`、`$vuetify.breakpoint` 等 Vuetify 专属 API，迁移困难 |
| 响应式方案混乱 | 通过 `$vuetify.breakpoint.name` 的 switch-case 硬编码断点，Magic Numbers 充斥代码 |
| CSS 碎片化 | 7 个独立 CSS 文件（5 个 flexible_*.css + global.css + verifier.css），维护困难 |
| 无统一布局 | 每个页面自己包裹 `<v-app>`，没有 `layouts/default.vue` |
| 统计方案落后 | 使用 CNZZ 统计，功能有限且已停止维护 |

### 1.2 重构目标

将 EasyBlog 从「后端 API 驱动的动态博客」重构为「GitHub 驱动的静态博客」，实现：

1. **文章数据 GitHub 化**：文章以 Markdown 文件形式存储在独立的 GitHub 仓库，通过目录结构组织
2. **Webhook 自动更新**：GitHub 仓库更新后通过 Webhook 触发 Vercel 自动重新构建和部署
3. **静态站点生成（SSG）**：Nuxt 全量预渲染为静态 HTML，部署到 Vercel
4. **TailwindCSS 替代 Vuetify**：统一使用 TailwindCSS 作为样式方案，移除所有 Vuetify 依赖
5. **Umami 数据统计**：接入 Umami 实现隐私友好的访问统计，替代 CNZZ

## 二、架构设计

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    GitHub 文章仓库（easyblog-articles）            │
│                                                                  │
│  articles/{YYYY}/{slug}.md    ← 文章 Markdown 文件               │
│  pages/*.md                   ← 独立页面（关于、友链等）           │
│  config.json                  ← 站点全局配置                      │
│  assets/images/               ← 文章配图                          │
└──────────────────────┬──────────────────────────────────────────┘
                       │ git push
                       ▼
┌─────────────────────────────────────────────────────────────────┐
│              GitHub Webhook → Vercel Deploy Hook                 │
│                                                                  │
│  文章仓库 push → 触发 Vercel Deploy Hook URL                     │
│  Vercel 自动 git clone 文章仓库 + 博客源码仓库                    │
│  执行 nuxt generate → 部署静态文件到 Vercel Edge Network         │
└──────────────────────┬──────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────────┐
│                        Vercel Edge Network                       │
│                                                                  │
│  ┌──────────────────┐   ┌──────────────────┐                    │
│  │ 静态 HTML/CSS/JS  │   │ Umami Analytics  │                    │
│  │ (全球 CDN 加速)   │   │ (访问统计埋点)    │                    │
│  └──────────────────┘   └──────────────────┘                    │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 数据流

```
作者编写文章 → git push 到文章仓库
  → GitHub Webhook 触发 Vercel Deploy Hook
  → Vercel 拉取文章仓库 + 博客源码
  → 执行 nuxt generate（SSG）
  → 部署静态文件到 Vercel Edge Network
  → 用户访问（全球 CDN 加速）
  → Umami 统计埋点上报
```

### 2.3 仓库关系

项目涉及两个 GitHub 仓库：

| 仓库 | 用途 | 说明 |
|------|------|------|
| `easyblog-web` | 博客源码 | Nuxt 2 项目代码，包含页面模板、组件、样式 |
| `easyblog-articles` | 文章内容 | 纯 Markdown 文件，不含代码逻辑 |

两个仓库独立维护。博客源码仓库通过构建时 clone 或 git submodule 的方式引用文章仓库。

## 三、文章仓库规范

### 3.1 仓库结构

```
easyblog-articles/
├── README.md
├── config.json              # 站点全局配置
├── articles/                # 文章目录
│   └── {YYYY}/
│       └── {slug}.md
├── pages/                   # 独立页面
│   ├── about.md
│   └── friends.md
└── assets/                  # 文章配图
    └── images/
        └── {YYYY}/
            └── {slug}/
                └── cover.png
```

### 3.2 文章 Front Matter 规范

每篇 Markdown 文件头部必须包含 YAML Front Matter，字段说明：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | string | ✅ | 文章标题 |
| `slug` | string | ✅ | URL 友好标识符，全局唯一 |
| `date` | string | ✅ | 发布日期，格式 `YYYY-MM-DD` |
| `updated` | string | ❌ | 最后更新日期，格式 `YYYY-MM-DD` |
| `category` | string | ❌ | 文章分类 |
| `tags` | string[] | ❌ | 标签列表 |
| `summary` | string | ❌ | 文章摘要，不填则构建时自动截取正文前 200 字 |
| `cover` | string | ❌ | 封面图路径，相对于仓库根目录 |
| `draft` | boolean | ❌ | 是否为草稿，默认 `false`，草稿不发布 |
| `featured` | boolean | ❌ | 是否为精选文章，默认 `false`，精选文章在首页轮播展示 |

### 3.3 config.json 站点配置

站点全局配置文件，包含以下配置项：

- **site**：站点标题、副标题、描述、语言、域名、作者信息（名称、邮箱、头像、简介）
- **theme**：主题色、字体族
- **features**：功能开关（评论、搜索、RSS、Sitemap）
- **navigation**：导航菜单项（首页、归档、分类、标签、关于）
- **footer**：版权信息、ICP 备案号、底部链接
- **umami**：Umami 统计配置（website_id、script_url）

## 五、Vercel 部署方案

### 5.1 部署架构

Vercel 是一个面向前端项目的静态站点托管平台，提供全球 CDN 加速、自动 HTTPS、Deploy Hook 等功能。EasyBlog 重构后将完全部署在 Vercel 上。

### 5.2 部署流程

1. **连接 GitHub 仓库**：在 Vercel 控制台中导入 `easyblog-web` GitHub 仓库
2. **配置构建设置**：
   - Framework Preset：Nuxt.js
   - Build Command：`npm run generate`
   - Output Directory：`dist`
   - Install Command：`npm install`
3. **配置环境变量**：在 Vercel 项目设置中添加 Umami 统计所需的环境变量
4. **配置 Deploy Hook**：在 Vercel 项目设置中创建 Deploy Hook URL
5. **配置 GitHub Webhook**：在文章仓库 `easyblog-articles` 中设置 Webhook，URL 指向 Vercel Deploy Hook

### 5.3 自动更新触发机制

当文章仓库有新的 git push 时：

1. GitHub Webhook 向 Vercel Deploy Hook URL 发送 POST 请求
2. Vercel 收到请求后自动触发重新构建
3. Vercel 执行构建流程：拉取最新文章仓库 → 解析 Markdown → `nuxt generate` → 部署
4. 部署完成后，新文章自动上线

### 5.4 构建时文章拉取策略

在 `nuxt.config.js` 的 `generate` 配置中，通过构建钩子在 `nuxt generate` 执行前拉取文章仓库：

- 使用 `git clone` 将文章仓库克隆到项目 `content/` 目录
- 如果 `content/` 目录已存在，则执行 `git pull` 增量更新
- 构建完成后，`content/` 目录中的文章数据被解析并渲染为静态页面

### 5.5 Vercel 配置要点

- Vercel 自动识别 Nuxt.js 项目，无需额外配置 Framework Preset
- 需要在 Vercel 项目设置中配置 Node.js 版本为 18.x
- 需要在 Vercel 项目设置中添加对文章仓库的读取权限（如果文章仓库为私有仓库，需配置 Deploy Key 或 Personal Access Token）
- Vercel 免费套餐支持 100GB 带宽/月，个人博客完全够用

## 六、Nuxt SSG 配置要点

### 6.1 核心配置

- `target: 'static'`：启用静态站点生成模式
- `ssr: true`：保留服务端渲染能力（generate 时需要）
- `generate.fallback: '404.html'`：未匹配路由回退到 404 页面
- `generate.routes`：动态生成所有文章的路由，从文章仓库的 manifest.json 或直接扫描 content/ 目录获取所有 slug

### 6.2 文章加载机制

创建一个 Nuxt 插件，在应用初始化时：

1. 扫描 `content/articles/` 目录下所有 `.md` 文件
2. 使用 `front-matter` 库解析每个文件的 YAML Front Matter 和正文内容
3. 过滤掉 `draft: true` 的草稿文章
4. 按 `date` 字段倒序排列
5. 将文章数据注入到 Vue 原型（`$articles`）和 Vuex Store 中
6. 所有页面和组件通过 `this.$articles` 或 Store 访问文章数据

### 6.3 动态路由生成

在 `nuxt.config.js` 的 `generate.routes()` 函数中：

1. 读取文章仓库中的所有 `.md` 文件
2. 提取每个文件的 `slug` 字段
3. 返回路由数组：`['/article/slug-1', '/article/slug-2', ...]`
4. Nuxt 在 generate 阶段会为每个路由生成对应的静态 HTML 文件

### 6.4 文章详情页路由变更

- 原路由：`/article/:index`（使用文章的数字 code 作为参数）
- 新路由：`/article/:slug`（使用文章的 slug 字符串作为参数）
- 页面文件从 `pages/article/_index.vue` 改为 `pages/article/_slug.vue`

## 七、Umami 统计接入

### 7.1 接入方式

Umami 提供两种使用方式：

- **Umami Cloud**（推荐）：在 umami.is 注册免费账号，获取 Website ID 和 Script URL，无需自建服务
- **自托管**：通过 Docker 在自己的服务器上部署 Umami 服务

对于个人博客，推荐使用 Umami Cloud 免费方案。

### 7.2 集成方式

在 `nuxt.config.js` 的 `head.script` 配置中添加 Umami 追踪脚本：

- 脚本属性：`async`、`defer`
- `data-website-id`：从 Umami 控制台获取的 Website ID
- `src`：Umami 脚本 URL（Umami Cloud 或自托管地址）

Website ID 和 Script URL 通过环境变量配置，不在代码中硬编码。

### 7.3 环境变量配置

在 Vercel 项目设置中添加以下环境变量：

- `UMAMI_WEBSITE_ID`：Umami 网站 ID
- `UMAMI_SCRIPT_URL`：Umami 脚本 URL

### Phase 5：部署配置

#### 指令 5.1：创建 Vercel 部署配置文件

```
请创建 vercel.json 文件（如果需要自定义 Vercel 配置），内容：

1. 配置 Clean URLs（自动将 /about 映射到 /about.html）
2. 配置 Headers：设置静态资源缓存策略
   - 图片资源：Cache-Control: public, max-age=31536000, immutable
   - CSS/JS 资源：Cache-Control: public, max-age=31536000, immutable
   - HTML 页面：Cache-Control: public, max-age=0, must-revalidate
3. 不需要配置 rewrites 或 redirects（Nuxt 自动处理路由）
```

#### 指令 5.2：配置 Vercel 项目

```
请在 Vercel 控制台完成以下配置：

1. 导入 easyblog-web GitHub 仓库到 Vercel
2. Framework Preset 选择 Nuxt.js
3. Build Command：npm run generate
4. Output Directory：dist
5. Install Command：npm install
6. 环境变量：
   - UMAMI_WEBSITE_ID：你的 Umami Website ID
   - UMAMI_SCRIPT_URL：你的 Umami Script URL
7. 在 Vercel 项目 Settings → Git → Deploy Hooks 中创建一个 Deploy Hook，
   命名为 "Article Update"，复制 Hook URL
```

#### 指令 5.3：配置 GitHub Webhook

```
请在 easyblog-articles 文章仓库中配置 Webhook：

1. 进入文章仓库 Settings → Webhooks → Add webhook
2. Payload URL：粘贴 Vercel Deploy Hook URL
3. Content type：application/json
4. Secret：不需要（Vercel Deploy Hook 不需要签名验证）
5. 事件类型：Just the push event
6. Active：勾选
7. 保存 Webhook
```

#### 指令 5.4：配置 Umami 统计

```
请完成 Umami 统计配置：

1. 在 umami.is 注册账号（或自托管 Umami）
2. 添加网站：输入 easyblog.top
3. 获取 Website ID 和 Script URL
4. 将 Website ID 和 Script URL 配置到 Vercel 环境变量中
5. 重新部署 Vercel 项目使环境变量生效
6. 访问网站，在 Umami 控制台验证统计数据是否正常上报
```

## 十、设计风格规范

### 10.1 配色方案

| 用途 | 颜色值 | TailwindCSS Class |
|------|--------|-------------------|
| 主色调 | #1e80ff | bg-[#1e80ff] / text-[#1e80ff] |
| 主色调悬停 | #1171ee | hover:bg-[#1171ee] / hover:text-[#1171ee] |
| 页面背景 | #f4f5f5 | bg-[#f4f5f5] |
| 卡片背景 | #ffffff | bg-white |
| 正文颜色 | #333333 | text-gray-800 |
| 次要文字 | #8a919f | text-gray-500 |
| 边框颜色 | #e4e6eb | border-gray-200 |
| 成功/通过 | #52c41a | text-green-500 |
| 警告 | #faad14 | text-yellow-500 |
| 错误/危险 | #ff4d4f | text-red-500 |

### 10.2 字体层级

| 层级 | TailwindCSS Class | 用途 |
|------|-------------------|------|
| 页面标题 | text-2xl font-bold | 各页面主标题 |
| 区块标题 | text-xl font-semibold | 区块标题（如"最新文章"） |
| 文章标题 | text-lg font-semibold | 文章列表中的标题 |
| 正文 | text-base | 文章正文、段落文字 |
| 辅助信息 | text-sm text-gray-500 | 日期、标签、作者信息 |
| 小字 | text-xs text-gray-400 | 版权、备案信息 |

### 10.3 布局规范

- 最大内容宽度：max-w-6xl（1152px），居中 mx-auto
- 首页双栏：左侧 lg:w-3/4，右侧 lg:w-1/4（小屏右侧隐藏）
- 文章详情双栏：左侧 lg:w-3/4，右侧 lg:w-1/4
- 卡片圆角：rounded-lg（8px）
- 卡片阴影：shadow-sm，悬停 shadow-md transition-shadow
- 页面水平内边距：px-4 sm:px-6 lg:px-8
- 区块垂直间距：my-6 或 py-6

### 10.4 响应式断点

| 断点 | 最小宽度 | 前缀 | 适用场景 |
|------|----------|------|----------|
| 默认 | 0px | （无前缀） | 手机竖屏 |
| sm | 640px | sm: | 手机横屏 |
| md | 768px | md: | 平板 |
| lg | 1024px | lg: | 桌面端（侧边栏显示/隐藏分界点） |
| xl | 1280px | xl: | 大屏桌面 |
